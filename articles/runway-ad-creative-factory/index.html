Ad Creative Factory: анатомия продукта

Разбор product shape: почему buyer, value unit, pricing и moat собраны именно так — и как это меняет economics performance-маркетолога.

Мы привыкли смотреть на AI-продукты через endpoint’ы. Открываешь документацию Runway — text_to_video, image_to_video, video_transform. Каждый с параметрами и response object. Продукт — правильная интеграция. Вызвал, получил результат, обработал rate limits. Цена продукта — цена credits. Конкуренты — те, у кого похожие endpoint’ы. Рынок — все, у кого есть API-ключ. Так выглядит endpoint-центричная модель — настолько привычная, что мы перестаём замечать, где она кончается и начинается продукт.

Product shape, который мы построили, переворачивает эту модель в каждом измерении. Buyer — один performance-маркетолог с трёхслойной работой, а не «любой с API-ключом». Value unit — креатив, готовый к запуску в рекламный кабинет: видеофайл, thumbnail, caption suggestions, targeting hints — а не task.output[0]. Конкуренция — против студии за $300–500 за креатив, а не против Pika и Kling. Pricing — пакетами $49/$149/$499, а не per-API-call. Каждое измерение endpoint-центричной модели перевёрнуто.

Кажется, что это просто надстройка над Runway API — API-wrapper с удачным позиционированием. Orchestration layer, который нельзя скопировать покупкой того же API. Buyer knowledge, превращающая «генератор видео» в замену агентства. Unit economics $0.75 за креатив против $500 в студии — разница, меняющая P&L покупателя, а не дающая скидку. На самом деле за каждым решением стоит трёхслойная продуктовая логика — и мы проследим её от endpoint’а до шести осознанных границ.

API — сырьё, не продукт

Runway API состоит из трёх endpoint’ов. text_to_video принимает текстовое описание и возвращает видеофайл. image_to_video — изображение на входе, видео на выходе. video_transform — существующее видео трансформируется в другой стиль или формат. Каждый endpoint параметризован: prompt, длительность до 10 секунд для Gen 4.5, соотношение сторон 16:9, модель. Каждый вызов стоит credits. Response object содержит URL готового файла. Для программиста это три завершённые точки входа — вызвал, получил, обработал.

В endpoint-центричной картине мира этим всё и исчерпывается. User — любой с API-ключом. Value — то, что вернул response object. Продуктовая работа сводится к правильной интеграции: обработка response objects, rate limiting, retry-логика, подсчёт credits. Каждый endpoint решает свою задачу целиком.

API действительно возвращает ровно то, что задокументировано. text_to_video генерирует видео по тексту. image_to_video — по изображению. video_transform — трансформирует существующее. Всё работает. Проблема не в том, что API возвращает не то. Проблема в том, что API возвращает capability — способность произвести видеофайл. Продукт — это не capability. Продукт — это решение buyer job, построенное на capability. Кирпичный завод производит кирпичи. Дом — не кирпич. Дом — проект, кладка, коммуникации. API — кирпичный завод. Он даёт сырьё. Продукт — это то, что строится из сырья под конкретную работу конкретного человека. API возвращает capability — способность сгенерировать видео, трансформировать видео. Capability — не решение. Понимание этой разницы — точка, с которой начинается продукт.

Buyer с трёхслойной работой

Runway API открыт для всех. Любой с API-ключом может вызвать text_to_video и получить видеофайл. Кажется, продукт на таком API должен быть для всех, кому нужно видео. Чем шире продукт — тем больше рынок.

Разным buyer нужны разные вещи. Creative Pro требуется интерактивный инструмент для creative exploration. E-commerce Team — интеграция с каталогом товаров и пакетная генерация под товарные позиции. Content Studio — серийный контент со стабильными персонажами и сквозным стилем. Enterprise — compliance, approval flows, multi-user управление. Каждый тянет за собой архитектурные требования, несовместимые с решениями для других. Продукт, пытающийся закрыть всех четырёх, не закроет ни одного.

Максиму 32 года. Performance-маркетолог в D2C-компании. Бюджет на трафик — $15 000 в месяц: Facebook Ads, TikTok, YouTube. Команда — он сам и один дизайнер на аутсорсе. KPI: CPA, CTR, ROAS. Каждый день без свежего креатива — падение ROAS, которое он видит в отчётах. Текущее решение — студия на аутсорсе: $300–500 за креатив, 3–5 дней на партию. Остальное — Canva для простых адаптаций и старые креативы с новыми заголовками. Креативы выгорают за 3–4 дня.

Работа, которую Максим нанимает продукт выполнить, трёхслойна. Functional job: «Когда нужна партия свежих креативов — получить 30–50 вариаций за час, а не 5 за неделю — чтобы запустить кампанию вовремя и не терять бюджет на выгоревших креативах». Emotional job: «Чувствовать контроль над creative pipeline — не быть заложником студии, чья скорость равна моей скорости». Social job: «Выглядеть как маркетолог, который масштабирует кампании быстрее конкурентов — мы уже тестируем пятую итерацию, пока они ждут студию».

Три слоя означают: продукт покупают не за генерацию видео. Functional — за скорость. Emotional — за контроль. Social — за статус. Только functional слой можно закрыть API-вызовом. Emotional и social требуют архитектуры, построенной от buyer, а не от endpoint’а.

Глубокое знание одного buyer даёт точность, недоступную при попытке закрыть всех. Мы знаем, что Максиму нужны пакеты креативов, а не штучные. Batch, а не interactive. Pricing $49–499 в месяц, а не per-API-call. Это не предположения — это выводы из трёхслойной работы. Продукт строится под конкретную работу Максима — три слоя, каждый требует своего ответа от продукта. Без buyer knowledge продукт становится «генератором видео для всех» — и его никто не покупает, потому что «для всех» означает «ни для кого».

Workflow: бриф → генерация → вариации → экспорт

Buyer и его работа ясны. Один API-вызов не решает задачу Максима. Но кажется, что orchestration layer — это просто несколько API-вызовов в правильном порядке. Зачем четыре компонента, если Максиму нужно «просто видео»?

API-вызов даёт видеофайл. Но путь от «нужна кампания» до «кампания готова к запуску» включает шаги, которые ни один endpoint не делает. Максиму нужен не prompt, а бриф — структурированное описание платформы, аудитории, продукта, tone of voice. Нужны не штучные видео, а партия креативов с N вариациями под разные сегменты. Нужен не набор файлов, а пакет, готовый к экспорту в рекламный кабинет. Четыре компонента закрывают каждый из этих разрывов.

Creative Generator — ядро. Принимает бриф из Brief Interface и создаёт креативы через Media Generation: text_to_video и image_to_video. В отличие от наивной интеграции, Generator получает на вход не сырой prompt, а структурированный бриф, параметры которого транслируются в вызовы API.

Variation Engine — второе ядро. Берёт созданный креатив и производит N вариаций через Video Transformation: смена стиля, адаптация под вертикальные форматы, стилистические варианты для A/B-тестирования. Одна идея → десять креативов под разные аудиторные сегменты. Закрывает functional JTBD: 50 вариаций из одной концепции вместо 5 заказов в студии.

Brief Interface — интерфейсный слой, заменяющий бриф-менеджера студии. Маркетолог заполняет не prompt для AI, а бриф для кампании: платформа, целевая аудитория, tone of voice, продукт. Brief Interface транслирует это в параметры генерации. Закрывает emotional JTBD: контроль над pipeline переходит от студии к маркетологу — через структурированный ввод, дающий предсказуемый результат.

Campaign Packager — выходной слой. Группирует сгенерированные креативы под кампанию, добавляет thumbnail, caption suggestions, targeting hints, собирает в экспортируемый пакет. Закрывает functional JTBD: результат — не набор файлов, а кампания, готовая к загрузке.

Realtime Interaction исключён осознанно. Максиму не нужен диалог с AI — нужен результат. Batch-режим сквозной: бриф → генерация → вариации → экспорт. Асинхронный pipeline, не диалог. Emotional JTBD закрывается через Brief Interface и предсказуемость pipeline, а не через интерактивность.

Конвейер завода — ещё не архитектурное бюро. Бюро берёт кирпичи и превращает их в дом по чертежу. Четыре компонента образуют workflow — не обёртку над endpoint’ами, а pipeline, где каждый шаг закрывает конкретный слой buyer job: Brief Interface — emotional JTBD, Campaign Packager — functional, Creative Generator и Variation Engine — оба слоя вместе.

Креатив, готовый к запуску — не видеофайл

API возвращает видеофайл. Response object содержит URL или бинарные данные. Endpoint отработал — задача выполнена. Кажется, продукт продаёт видеофайлы. Thumbnail, caption, targeting hints — косметика поверх. Главное — генерация.

Максим заходит в рекламный кабинет Facebook Ads чтобы запустить кампанию. Он загружает креатив. Креатив — не видеофайл. Это видеофайл плюс thumbnail, который пользователь увидит до автовоспроизведения. Плюс caption — текст, усиливающий сообщение видео. Плюс targeting hints — понимание, под какой интерес-сегмент креатив заточен. Ни один endpoint Runway API не возвращает такой составной объект. task.output[0] — это видеофайл. Thumbnail нужно сгенерировать, caption — сформировать на основе prompt’а и контекста кампании, targeting hints — вывести из брифа. Видеофайл — только один из четырёх элементов.

Теперь масштаб. Один API-вызов даёт один результат. Кампания требует не одного креатива. Креативы выгорают за 3–4 дня, ROAS падает — нужна партия: 30, 40, 50 вариаций под разные сегменты аудитории, под разные площадки. API не знает про кампанию. API не знает про аудиторные сегменты. API знает про prompt и credits.

Такой value unit не даётся бесплатно. Campaign Packager потребовался именно для этого — компонент постобработки, собирающий разрозненные outputs API в единый объект, готовый к использованию. Дополнительная логика поверх API — плата за то, чтобы продавать не муку, а хлеб.

Теперь value proposition против студии встаёт на измеримую основу. Пять pain relievers: скорость 10x (50 креативов за час против 5 за неделю), цена 400–600x ($0.75 против $300–500), Video Transformation вместо новых заказов (креатив нельзя изменить — студия требует новый заказ и новое ожидание), отсутствие коммуникационного overhead (бриф в интерфейсе → результат, без цепочки уточнений), A/B-тестирование (одна идея → 10 стилистических вариаций). Четыре gain creators: скорость итерации (пять циклов тестирования за неделю вместо одного), гиперсегментация (каждый сегмент получает свой креатив с уникальными targeting hints), data-driven creative (все креативы в одном pipeline → атрибуция), масштабирование без найма (fixed cost продукта против variable cost студии).

Мука — не хлеб. Пекарня не продаёт муку — продаёт хлеб. Мука — сырьё, требующее замеса, расстойки, выпечки. Хлеб — продукт, готовый к употреблению. Value unit — креатив, готовый к запуску: видеофайл + thumbnail + caption suggestions + targeting hints. Это то, что Максим загружает в рекламный кабинет, и то, за что он платил студии $300–500. Без постобработки продукт не делает работу buyer — он только генерирует сырьё.

Пакеты, не credits — pricing как бизнес-модель buyer

Когда разработчик думает о pricing AI-продукта, модель очевидна. Берём стоимость credits, добавляем наценку — вот цена. Цена продукта равна цене API-вызова плюс маржа. Креативы стоят по-разному — credits обеспечивают гибкость.

Максим не считает credits. Когда он планирует кампанию, он думает: «Мне нужно 20 креативов под новую коллекцию к среде». Его mental model — креативы на кампанию, а не credits на вызов. Pricing, не совпадающий с mental model buyer, делает продукт чужим, сколько бы он ни стоил. Маркетолог вынужден переводить бизнес-единицу в техническую — без гарантии, что результат будет приемлемого качества с первой попытки. Предсказуемость затрат исчезает.

Пакетная модель совпадает с тем, как Максим думает о кампаниях. Три тира: Starter — $49 в месяц, 5 пакетов, до 50 креативов, базовые вариации. Growth — $149 в месяц, 20 пакетов, до 200 креативов, расширенные вариации. Agency — $499 в месяц, 100 пакетов, до 1000 креативов, A/B-тестирование. Каждый тир включает генерацию из брифа, вариации и экспорт. Разница — в объёме и дополнительных возможностях.

Теперь unit economics со стороны buyer. Growth-тир: 200 креативов за $149 — около $0.75 за креатив. Студия: $300–500 за креатив. Разница — 400–600 раз. По скорости: 50 креативов в час против 5 в неделю — десятикратная. По времени ожидания: минуты против 3–5 дней — три порядка. 400–600x — это не скидка. Это смена economics бизнеса покупателя.

Когда креатив стоит $0.75, а не $500, маркетолог может позволить себе тестировать гипотезы. «А что если другой стиль?» — в студии новый заказ, новые деньги, новое ожидание. В продукте — одна вариация в Variation Engine. A/B-тестирование перестаёт быть роскошью и становится операционным процессом.

С нашей стороны. Cost structure на Growth-тир: 200 креативов × 125 credits, Unlimited-тир Runway API — около $7 API-затрат. Инфраструктура — примерно $20 в месяц. $149 revenue минус $27 cost — $122 маржи, 82%. Маржа высокая, но модель имеет напряжение: Agency-тир при 1000 креативов даёт 90% маржи, при 5000 — падает до 50%. Пакетная модель не масштабируется линейно с power user. Предсказуемость для маркетолога означает revenue cap на одного клиента — напряжение, которое потребует overage pricing в v2.

Такси не продаёт бензин по литрам. Оно продаёт поездку. Пассажиру не важно, сколько бензина сожжёт двигатель — важна цена поездки. Пакетная модель совпадает с mental model Максима: «кампания = N креативов». Unit economics $0.75 за креатив против $500 студия — это не скидка, это смена economics бизнеса покупателя. Pricing — не способ взять деньги, а moat: конкурент на API-ценообразовании не даст таких цифр.

Структурный moat — три слоя защиты

Когда мы проектируем orchestration layer и анализируем JTBD, конкуренты с простым API-wrapper могут уже быть на рынке. Если конкурент купит тот же Runway API и напишет свой workflow — он скопирует продукт. Moat, построенный на «мы были первыми» или «у нас лучший доступ к API», исчезает в момент, когда Runway открывает API следующему конкуренту.

Ad Creative Factory строит структурную защиту. Три слоя, каждый из которых не зависит от эксклюзивности доступа к моделям.

Слой 1: workflow над endpoint’ами. Конкурент может купить тот же Runway API — text_to_video, image_to_video, video_transform. Но четыре компонента не равны трём endpoint’ам. Brief Interface структурирует ввод в параметры генерации. Variation Engine производит N вариаций из одного креатива — операция, отсутствующая в API как отдельный продукт. Campaign Packager собирает составной value unit из разрозненных outputs. Копирование требует воспроизведения всего pipeline — архитектурной работы, а не покупки доступа. Плата: больше точек отказа, архитектура вместо скрипта. Но именно это превращает набор endpoint’ов в продукт.

Слой 2: buyer knowledge над feature list. Конкурент, начинающий с API, строит feature list: text_to_video, image_to_video, video_transform. Это генератор видео для всех. «Для всех» означает «ни для кого» — продукт не решает конкретную работу конкретного человека. Мы знаем, что Максиму нужны пакеты, batch, pricing $49–499. Это знание — результат трёхслойного JTBD-анализа. Конкурент без него сделает продукт, не совпадающий с mental model buyer. Плата: buyer knowledge не масштабируется на другие сегменты без повторного анализа.

Слой 3: pricing model над API-pricing. Pay-per-package даёт unit economics $0.75 за креатив для buyer при marginal API cost около $0.05. Конкурент на per-API-call pricing не может показать такие цифры — его цена привязана к credits, которые не транслируются в «креатив на кампанию». Пакетная модель создаёт ценность через предсказуемость, но ограничивает revenue на одного клиента — плата за moat.

Moat строится инкрементально. V1 — минимальный viable moat: workflow обязателен как определение продукта, buyer knowledge покрывает 80% Максима, pricing — простая пакетная структура. V2 — углубление: Ad Platform API, Realtime как premium-режим, расширение buyer на E-commerce и Content Studio.

Замок из песка строится быстро. Первый дождь смывает стены. Каменный замок требует времени, но стоит веками. Три слоя moat — структурные, не маркетинговые. Слой 1 делает копирование архитектурной задачей. Слой 2 превращает отсутствие buyer knowledge в конкурентный дефект. Слой 3 создаёт unit economics, невозможные при API-ценообразовании. Продукт нельзя скопировать покупкой того же Runway API — каждый слой делает копирование дороже.

Конкуренция — против студии, не против Pika

Когда разработчик смотрит на рынок AI-видео, он видит Pika и Kling. У них тоже video generation. Конкуренция — по качеству модели и цене credits. Кто дешевле за API-вызов и у кого лучше модель — тот и выиграл.

Максим не выбирает между AI-инструментами. Он выбирает между способами закрыть свою работу. Студия на аутсорсе: $300–500 за креатив, 3–5 дней на партию, 5 креативов в неделю — не масштабируется. Canva/DIY: требует навыков дизайна, не производит видео-креативы. «Ничего не делать»: выгорание креативов → падение ROAS → потеря бюджета на трафик — измеримый убыток. Каждая альтернатива — другой способ решить ту же работу, и каждая имеет измеримый дефект: цена, скорость или качество.

Продукт не играет в игру «чей AI лучше». Конкуренция — против альтернативных способов решения. Нарратив — «замена агентства», не «генератор видео».

Другие AI-инструменты решают другую задачу. Pika и Kling — consumer-инструменты для creative exploration. У них нет B2B-фокуса: нет workflow под production, нет buyer knowledge про performance-маркетолога, нет пакетов под кампании. Их buyer — creative pro, исследующий идеи. Наш buyer — маркетолог, производящий креативы под конкретную кампанию.

Конкурентное framing против студии создаёт рынок там, где feature-конкуренты его не видят. Маркетолог сравнивает не продукт с Pika — он сравнивает продукт со студией. Другой рынок — performance marketing, а не AI video generation. Другая готовность платить — $49–499 в месяц за production-пайплайн, а не $10 за credits. Другой decision process — ROAS-driven, а не feature comparison. Такси не конкурирует с производителями автомобилей. Пассажиру не важно, кто сделал машину — важно доехать. Продукт конкурирует против альтернативных способов решения работы Максима — студия, Canva, «ничего не делать». Это другой рынок — performance marketing, другая готовность платить, другой decision process. Максим не сравнивает продукт с Pika — он сравнивает со студией.

Осознанные границы — не баги, а форма

Глядя на продукт снаружи, разработчик видит ограничения. 5–10 секунд — технический лимит API, не продуктовое решение. Realtime нет — просто не успели добавить. Batch-only — потому что не сделали интерактивный режим. Отсутствие фичи выглядит как упущение. Границы — баги, которые починят в следующих версиях.

Ad Creative Factory имеет шесть осознанных границ. Каждая — scope decision, а не дефект.

5–10 секунд. Не ограничение, которое надо обойти. Определение ниши: Stories, Reels, Shorts, in-feed video — форматы, в которых работает performance-маркетинг. Длинное видео — другой продукт для другого buyer.

16:9 как основной формат. Gen 4.5 производит горизонтальное видео. Адаптация под вертикальные форматы — через Variation Engine и Video Transformation. Генерация в одном формате, вариации в нескольких.

Без гарантии стабильности персонажа. Каждый креатив — самостоятельная единица. Продукт не для сериального контента, где персонаж должен выглядеть одинаково от эпизода к эпизоду. Исключает Content Studio как buyer для v1.

Content moderation. Только легальный контент. Продукт не для grey-area вертикалей, где moderation requirements принципиально другие.

Batch-only — без Realtime Interaction. Не «не успели». Осознанный scope cut: production mindset маркетолога требует результата, а не диалога. Emotional JTBD — контроль — закрывается через Brief Interface и предсказуемость pipeline.

Без Ad Platform Integration. Campaign Packager экспортирует креативы в файлы. Загрузка в рекламный кабинет Facebook/TikTok остаётся ручной. Интеграция с Ads API — v2, осознанный scope cut, фокусирующий первую версию на создании креативов.

Нож — не неудачная ложка. У ножа своя работа. Каждый constraint — осознанный выбор. 5–10 секунд определяет нишу Stories/Reels/Shorts. Batch-only — production mindset. Исключение Realtime и Ad Platform API — scope cut под v1. Границы формируют продукт: они говорят не «чего продукт не может», а «какую работу продукт делает и — осознанно — какую не делает».

Главный конфликт — moat против времени

Все элементы собраны: buyer, workflow, value unit, pricing, moat, конкурентное framing, границы. Но они вступают в конфликт. Три слоя moat плюс четыре компонента плюс шесть осознанных границ — это архитектура, требующая времени на разработку и тестирование. Пока мы строим orchestration layer, анализируем JTBD и проектируем пакетный pricing, конкуренты с простым API-wrapper уже на рынке. Кажется, скорость выхода важнее глубины защиты.

Главный конфликт продукта — между глубиной защиты и скоростью выхода на рынок. Он затрагивает большинство других решений в продукте.

Moat — не overengineering, а условие существования продукта. Без orchestration layer продукт превращается в API-wrapper, копируемый за недели. Без buyer knowledge — в генератор видео для всех, который никто не покупает. Без pricing model — в commodity, конкурирующую ценой credits. Три слоя не роскошь — они определение продукта.

Но три слоя плюс четыре компонента плюс шесть границ дают максимальную защиту при максимальном времени до рынка. Полная реализация всех слоёв до запуска — путь к тому, что конкуренты с простыми API-wrapper’ами захватят пользователей раньше.

Разрешение — не выбор «меньше moat — быстрее на рынок». Минимальный viable moat для v1: workflow обязателен — это определение продукта, не опция; buyer knowledge покрывает 80% глубины — даёт точное попадание в работу Максима; pricing — простая пакетная структура из трёх тиров. Ad Platform API, Realtime как premium и расширение buyer на E-commerce и Content Studio уходят в v2. Moat строится инкрементально: v1 доказывает product-market fit, v2 углубляет защиту.

Напряжение порождает связанные конфликты. Глубокий фокус на одном buyer исключает четыре других рынка из v1. Пакетная модель даёт предсказуемость для маркетолога, но Agency-тир при перегрузке теряет маржу — 90% при 1000 креативов, 50% при 5000. Каждый tradeoff — плата за выбранную форму продукта.

Фундамент небоскрёба не построить за неделю. Но без него башня рухнет при первом ветре. Главный конфликт — между глубиной защиты и скоростью выхода — разрешается не выбором «меньше moat — быстрее на рынок», а минимальным viable moat для v1: workflow обязателен, buyer knowledge покрывает 80% Максима, pricing — простая пакетная структура. V1 — фундамент: не весь небоскрёб, но то, на чём он стоит. Moat строится инкрементально — v1 доказывает product-market fit, v2 углубляет защиту и расширяет buyer.
